# 入职智引系统串行开发任务看板

## 全局环境基线（全员必读）

- Python 版本: 3.12.10 (严禁使用其他版本，避免依赖冲突)
- LLM 模型: qwen3.7-plus (通过百炼 API 调用)
- 代码风格：禁止使用emoji，使用中文注释与打印，各人员完成对应阶段后需编写DEVLOG.md
- Embedding 模型: text-embedding-v4
- 鉴权方式: 统一读取系统环境变量 API_KEY，代码中禁止硬编码
- 核心框架: LangChain + Pandas + NumPy + ChromaDB + FastAPI + Vue3

## 项目文件结构

```
OnboardingBot/
├── backend/
│   ├── data/raw/          # 原始 PDF/TXT/DOCX/MD 文件存放处
│   ├── data/rules/        # 结构化 CSV（包含薪资绩效表、用户表、日志表、系统配置表）
│   ├── VectorStore/       # Chroma 向量库本地目录
│   ├── src/tools/         # 计算函数与向量/文件管理原子函数
│   ├── src/core/          # LangChain 链路与 Prompt 模板
│   ├── src/api/           # FastAPI 接口路由
│   └── tests/             # 测试用例
├── frontend/              # Vue3 前端
└── docs/                  # 文档
```

## 阶段一：成员二（数据工程师）—— 知识资产与计算引擎构建

**阶段目标**：产出可直接被后端加载的向量库、结构化规则表（CSV）、以及整套本地资产管理的原子函数（增删改查 Python 函数）。

**前置条件**：无（项目起点）

- **1. 工程基座与环境验证**
  - **执行动作**：初始化 Git 仓库；配置 `.gitignore`。
  - **验收标准**：Git 仓库结构规范，`.gitignore` 配置正确。
- **2. 混合格式知识库构建（LangChain + ChromaDB）**
  - **执行动作**：编写 `build_vectorstore.py` 脚本。基础版升级：不仅用 `PyPDFLoader` 读取《员工手册》，还要编写简单的后缀判断逻辑，支持通过 `TextLoader` 读取 `.txt` / `.md`，或通过 `Docx2txtLoader` 读取 `.docx`。
  - **核心细节**：使用 `RecursiveCharacterTextSplitter (chunk_size=512, overlap=50)` 切片时，必须在文本块的 `metadata` 中强行写入 `source`（文件名）和 `page`（页码），为前端引用溯源留出完整数据。使用 `DashScopeEmbeddings(model="text-embedding-v4")` 向量化并写入本地 `./VectorStore/`。
  - **验收标准**：运行脚本后 `./VectorStore/` 目录生成，检索测试 Top-3 能返回带文件名和页码的结果。
- **3. 结构化计算引擎与基础本地数据库（Pandas）**
  - **执行动作**：
    - 用 `pdfplumber` 提取薪资/绩效表格，Pandas 清浅导出为 `data/rules/salary.csv`。编写 `calculate_deduction(salary, late_count)` 等纯计算函数。
    - **轻量本地持久化（替代方案）**：手工初始化或用 Pandas 创建基础管理表，包括 `data/rules/users.csv`（存放用户名、密码等基础信息）、`data/rules/logs.csv`（初始为空，用于存放对话日志）、`data/rules/config.csv`（存放文本拆分参数、接口超时时间等系统参数）。
    - 编写 `pytest` 单元测试。
  - **验收标准**：所有基础 CSV 文件无缺失，`pytest` 全部通过，计算函数可供组长直接调用。
- **4. 动态资产管理函数编写（核心补充）**
  - **执行动作**：在 `src/tools/file_manager.py` 中编写基础 Python 工具函数，供组长后续封装 API 调用：
    - `delete_doc_vector(filename)`：根据文档名称从 Chroma 向量库中删除对应文档的所有向量数据。
    - `add_user(username, password, email, department)`：向 `users.csv` 追加写入一条新用户信息。
    - `save_chat_log(user_id, question, answer, kb_source)`：向 `logs.csv` 追加写入单条对话日志记录。
  - **验收标准**：编写的工具函数调用成功，能对本地 CSV 文件及本地向量库进行准确的增删与读写。
- **5. 阶段一交付**
  - **交付清单**：① `./VectorStore/` 目录 ② `data/rules/*.csv` 基表 ③ `src/tools/` 下的计算与管理原子函数。
  - **验收标准**：组长能成功加载向量库、读取和写入 CSV、调用计算与管理函数返回正确结果。

## 阶段二：组长（后端开发）—— RAG 引擎与 API 服务化

**阶段目标**：基于阶段一交付物，构建支持多轮对话、流式输出、Function Call 以及基础管理后台接口的 FastAPI 服务。

**前置条件**：阶段一交接验收通过

- **1. LangChain RAG 链路搭建**
  - **执行动作**：加载成员二交付的 `./VectorStore/`；初始化 `ChatTongyi(model="qwen3.7-plus")`；使用 LCEL 构建检索链；编写 HR 助手 Prompt Template，强制包含引用溯源指令（如要求回答末尾统一标注“【来源:文件名-第X页】”）；集成 `RunnableWithMessageHistory` 实现多轮对话记忆。
  - **验收标准**：终端测试多轮追问上下文关联正确，引用来源标记格式严格统一。
- **2. 基础管理功能 API 封装（利用成员二的工具函数）**
  - **执行动作**：
    - 封装基础登录认证接口：`POST /api/auth/login`，通过读取并匹配 `users.csv` 校验账号密码。
    - 封装文档资产管理接口：`GET /api/docs` 展示文档列表，`POST /api/docs/delete` 调用成员二的删除函数清理指定文件的向量与本地记录。
    - 封装对话日志查询接口：`GET /api/logs` 供前端读取 `logs.csv` 记录。
    - 对话路由集成：在 `POST /api/chat` 中加入基础逻辑，如果回答或提问触发敏感词，直接使用 Python 规则匹配并替换为 `*`；每次问答结束，调用日志函数存入 `logs.csv`。
  - **验收标准**：接口全量跑通，前端或 Postman 能够通过 API 正常完成登录、文档删除与日志读取。
- **3. Function Call 集成与流式接口封装**
  - **执行动作**：导入成员二的 `calculator.py`，用 `@tool` 封装并 `bind_tools` 到 LLM；使用 `astream_events + StreamingResponse` 实现 SSE 流式输出。读取 `config.csv` 的配置参数，若单轮答案生成超过限定长度，则在后端拦截并截断，返回友好提示。
  - **验收标准**：计算类问题精准回答；流式响应正常；单轮字数限制及超时中断控制生效。
- **4. 阶段二交付物打包与交接**
  - **执行动作**：整理交付清单：① 可运行的 FastAPI 服务 ② 完整接口文档（JSON Schema） ③ 基础 Mock 数据。
  - **交接验收标准**：成员三能用 Mock 数据跑通前端的基础对话与管理后台页面流程。

## 阶段三：成员三（前端开发）—— 交互实现与体验打磨

**阶段目标**：基于阶段二接口，构建包含“智能助手聊天”与“本地管理后台”的双版块交互界面。

**前置条件**：阶段二交接验收通过

- **1. 双版块前端基座搭建**
  - **执行动作**：Vue3 + Vite 初始化。设计基础的侧边栏导航布局，划分两个核心版块：
    - **版块一：AI 智能对话**（聊天窗口布局、Markdown 渲染组件、快捷提问按钮、历史会话控制）。
    - **版块二：本地管理后台**（包含文档列表表格、登录页、用户管理表、对话日志列表表格）。
  - **验收标准**：使用 Mock 数据能够完整切换和渲染对话与管理后台两个版块的各个页面。
- **2. SSE 流式对接与引用溯源高亮**
  - **执行动作**：对接真实 `/api/chat` SSE 接口实现打字机效果；实现会话清除与快捷提问点击自动发送。
  - **溯源交互**：前端通过正则解析后端返回的 `【来源:xxx】` 标记，将其渲染为高亮的可点击标签，点击后直接在侧边栏或弹窗中渲染展示对应的原始文本块内容。
  - **验收标准**：流式显示平滑；刷新页面历史对话管理正常；点击来源标签能顺畅展示出处原文。
- **3. Admin后台大盘交互对接**
  - **执行动作**：全面对接后端管理接口。
    - 在文档管理页面，对接删除接口，点击表格的“删除”按钮能联动后端执行向量清理并刷新列表。
    - 在对话日志页面，获取日志列表，利用前端计算属性或简单逻辑实现按用户 ID 或关键词的过滤筛选。
    - 对接登录界面，实现 Token 或基础登录状态的本地缓存。
  - **验收标准**：登录、文档列表删除、日志查看与条件筛选交互功能全部走通。
- **4. 阶段三交付**
  - **交付清单**：① 可构建的前端工程 ② 前端自测报告 ③ 基础运行部署说明。

## 串行模式风险管控要点
| **风险点**               | **应对措施**                                                 |
| ------------------------ | ------------------------------------------------------------ |
| **上游延期导致下游阻塞** | 阶段一和阶段二一旦在初期定好数据结构和接口定义（JSON Schema），前端（阶段三）立刻使用 Mock 数据编写页面，无需等待后端逻辑完全写完。 |
| **接口变更导致前端返工** | 阶段二接口交付后，若因业务逻辑需要增加或微调字段，必须第一时间书面通知成员三，并同步更新前端的 Mock 数据。 |
| **本地数据被污染或改坏** | 阶段一交付的原始 `rules/*.csv` 基础表在本地单独建立一个 `backup/` 文件夹。每次演示或彩排前，一键复制覆盖，确保环境重置干净。 |
